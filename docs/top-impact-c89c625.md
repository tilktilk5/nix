# Impact check: pulling `c89c625` onto `top`

**Date:** 2026-07-20
**Pull:** `3033bbf` → `c89c625` — *"air: standalone home-manager on the MacBook Air, hyprvtb fix, battery module"*
**Question asked:** does fast-forwarding `main` to this commit change anything about how the `top` machine builds or behaves?
**Answer:** No functional change to `top`. Verified below by comparing the two revisions' *built derivations*, not just reading the diff.

The commit is almost entirely second-machine bring-up: a new standalone
`homeConfigurations.air` output (Fedora Asahi MacBook Air) plus the `host`
plumbing to share `./lam.nix` between the two. `top` keeps `programs.hyprland`,
its NixOS layer, and its full package set.

## What the pull did

- `git pull --ff-only` on a clean tree: fast-forward only, no merge, no local
  work to clobber. The pull updates the working tree only — nothing is rebuilt
  or activated until a `rbsys`.

## How "no effect on `top`" was verified

All checks were done against the two git revisions directly (`git+file://…?rev=`),
so nothing on the live system was touched.

### 1. Built-system derivation compared old vs new

```
nix eval …#nixosConfigurations.top.config.system.build.toplevel.drvPath   # both revs
```

The two `toplevel` drvPaths differ — but `nix-diff` showed the differences are
entirely **home-managed config file text**, not packages or system units:
`hyprland.lua`, the new `host.lua`, and the quickshell battery scripts (details
below).

### 2. Resolved `home.packages` name list diffed (host = "top")

```
nix eval …#…home-manager.users.lam.home.packages --apply 'sort (map name)'   # both revs
```

Diff of the two fully-resolved lists showed exactly **one** delta:

- `hyprland-0.55.4` dropped from `home.packages`.

Every other package that `wm.nix` / `base.nix` / `net.nix` / `dev.nix` /
`utils.nix` / `misc.nix` / `game.nix` / `media/*` moved behind
`lib.optionals (host != "air")` / `isx86_64` / `host == "top"` comes right back
for `top` — the resolved set is otherwise identical.

### 3. The one package delta is inert on `top`

`hyprland` in `home.packages` was redundant: the compositor is provided
system-wide by `sys/dsk/hyprland.nix`:

```
programs.hyprland.enable = true   # unchanged — true in both 3033bbf and c89c625
```

Verified `config.programs.hyprland.enable == true` for `top` in both revs.
Dropping the duplicate from `home.packages` does not remove Hyprland from the
system.

### 4. Spot-checked the `breeze` → `kdePackages.breeze` swap in `kde.nix`

Both revisions resolve to the same `breeze-6.7.2` for `top`. No change.

### 5. `top`-branch values that stayed put

- `zsh.nix`: `update` / `rbsys` / `rbhome` / `trash` aliases identical on the
  `host == "top"` branch.
- `plasma.nix`: `kwinrc.Xwayland.Scale = 1` on `top` (the `== 2` only applies to
  `air`).
- `hyprvtb.nix`: `top` (host != "air") uses plain `pkgs`, i.e. the same
  `pkgs.callPackage ./hyprvtb {}` as before; the forced-`GIT_*`-unknown override
  is `air`-only.

## Runtime-file changes that *do* land on `top` (all no-ops here)

These are home-managed files whose content changed, so they update on the next
`rbsys` — but none change behavior on `top`:

- **`~/.config/hypr/host.lua`** — new, generated by `home/prog/hypr-host.nix` as
  `{ scale = "1", laptop = false }` for `top`. `hyprland.lua` is seed-once, so the
  live copy isn't overwritten anyway; even re-seeded it's `scale = "1"` (a no-op).
- **quickshell battery widget** (`SysInfo.qml`, `StatusPanel.qml`,
  `scripts/sysinfo.sh`) — regenerated on rebuild (not seed-once). The battery
  `Stat` is `visible: batteryPct >= 0`; with no `/sys/class/power_supply/BAT*`
  node on `top` it reports `-1` and stays hidden. Brightness still uses `ddcutil`
  (no `/sys/class/backlight` panel), same as before. Takes effect after a qs
  reload.

## Unrelated cleanup done at the user's request (not from the pull)

Removed the two hyprvtb runtime state files so `top`'s saved window layout is
reset:

- `~/.local/state/hyprvtb/session.tsv` (saved Meta+Ctrl+S session snapshot)
- `~/.local/state/hyprvtb/geometry.tsv` (per-window-class position memory)

Both are pure runtime state (not nix-managed); their load paths bail cleanly when
absent and recreate on next save.

## Conclusion

Fast-forwarding `top` to `c89c625` and rebuilding is safe: identical resolved
package set apart from a redundant `hyprland` entry already covered by
`programs.hyprland.enable`, and the only new runtime files are inert on this
host. Reboot / `rbsys` when ready.
